# 能力路线图：知识库 → MCP → Skill

本项目的差异化在于「翻译质量的可控性」。三期演进都围绕**同一条主线**：让术语与风格知识越来越结构化、越来越可复用、越来越自动化。

```
一期 结构化术语知识库   →   二期 MCP 检索接入   →   三期 Claude Skill 封装
本地、纯文本起步           知识外置、实时检索        翻译流水线被 AI 调度
```

---

## 一期 · 结构化术语知识库（✅ 已落地）

> 已实现：`app/knowledge.py` 结构化数据模型 + 服务端存储 + CRUD API（`/api/knowledge*`）+ 翻译按文档命中过滤注入（`stats.glossary_hits`）+ 前端术语表格编辑器 + 旧纯文本兼容导入。详见 [模块说明/知识库设计.md](../模块说明/知识库设计.md)。下文保留设计背景。


### 现状问题
当前知识库是一坨纯文本（`static/app.js` 存 `localStorage`，后端 `pdf2zh_engine.py` 直接拼进 prompt）。它**不是知识库**：
- 不能按文中出现的术语「命中」并精确替换，全靠模型自觉。
- 不能跨文档、跨设备复用与共享。
- 无版本、无审校、无冲突检测。

### 目标
把知识库升级为**结构化数据**：

```jsonc
{
  "name": "学术论文",
  "version": 2,
  "glossary": [
    { "src": "large language model", "dst": "大语言模型", "case_sensitive": false, "note": "" }
  ],
  "style_rules": ["使用正式流畅的学术中文", "保持论文语气"],
  "do_not_translate": ["模型名", "数据集名", "公式", "引用"]
}
```

### 关键改动
- **数据模型**：见 [公共设计/数据模型.md](../公共设计/数据模型.md) 的 `KnowledgeProfile`。
- **后端存储**：从浏览器 `localStorage` 迁到服务端 `storage/knowledge/*.json`，新增 CRUD API（见 [接入规范/HTTP-API.md](../接入规范/HTTP-API.md) 预留段）。
- **命中注入**：翻译前扫描文档文本，只把**实际出现**的术语注入 prompt（减少噪声、提升命中率），并可对禁译词做硬保护。
- **前端**：知识库视图改为「术语表格 + 风格列表 + 禁译列表」结构化编辑，替换当前 textarea。

### 验收
- 能新建/编辑/保存多套结构化知识库到服务端。
- 翻译时质检报告显示「本次命中 N 个术语」。
- 导入导出 JSON 兼容旧的纯文本（向后兼容）。

## 二期 · MCP 检索接入（✅ 已落地）

> 已实现：内置术语 MCP Server（`mcp_server/glossary_server.py`，FastMCP/stdio，工具 `glossary_search` / `style_get`）+ MCP Client（`app/mcp_client.py`）+ 引擎接线与**失败自动回退本地**。在「接入设置 → 知识来源」选 `MCP 检索` 启用。详见 [接入规范/MCP接入规范.md](../接入规范/MCP接入规范.md)。下文保留设计背景。

### 动机
当术语量大（上万条）、或需要多个工具/团队共享同一套术语时，把知识库塞进每次 prompt 不现实。改为**外置 + 实时检索**。

### 方案
- 起一个 **MCP Server**（术语/风格知识服务），暴露工具：
  - `glossary.search(text, top_k)` —— 给定段落，返回相关术语条目。
  - `style.get(domain)` —— 返回某领域风格规则。
- 本翻译服务作为 **MCP Client**，在翻译每页/每段前调用检索，把命中的术语动态注入 prompt。
- 知识库后端可由本地 JSON 升级为向量库（如 sqlite + 嵌入）以支持语义检索。

### 关键约定
详见 [接入规范/MCP接入规范.md](../接入规范/MCP接入规范.md)：工具 schema、调用时机、降级策略（MCP 不可用时回退到一期本地知识库）。

### 验收
- 配置一个 MCP endpoint 后，翻译流程能自动检索并注入术语。
- 断开 MCP 自动回退本地知识库，不阻塞翻译。

## 三期 · Claude Skill 封装（✅ 已落地）

> 已实现：`.claude/skills/translate-pdf/`（`SKILL.md` + `translate_pdf.py`）。把整条流水线封装成可被 Claude Code 调度的 skill，内部复用一/二期的 HTTP API，支持 argos/openai_compatible/mock 与 `--knowledge` / `--knowledge-source mcp`。详见 [接入规范/Skill接入规范.md](../接入规范/Skill接入规范.md)。下文保留设计背景。

### 动机
让整条「上传 → 文档级上下文抽取 → 知识库检索 → 翻译重排 → 质检」流水线，能被 Claude Code 这类 agent 直接调度，融入更大的 AI 工作流（例如「读这篇论文并翻译关键章节再总结」）。

### 方案
- 把翻译能力封装为 **skill**（`SKILL.md` + 触发说明 + 脚本入口），或暴露为可被 agent 调用的工具。
- skill 内部复用现有 HTTP API（一/二期成果），对外提供「翻译此 PDF / 用某知识库 / 返回质检结论」的高层接口。

### 关键约定
详见 [接入规范/Skill接入规范.md](../接入规范/Skill接入规范.md)：触发条件、输入输出契约、与 MCP 的协作关系。

### 验收
- 在 Claude Code 中通过 skill 触发一次完整翻译并拿到结构化结果。

---

## 依赖关系

- 二期依赖一期的**结构化数据模型**（没有结构化就没法检索）。
- 三期依赖一/二期的**稳定 HTTP API**（skill 只是上层编排，不重复造引擎）。
- 因此**严格按期推进**，每期都能独立交付价值，不必等到三期才可用。
