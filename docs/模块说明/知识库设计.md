# 知识库设计

知识库是本项目质量护城河，是三期路线图的主线。本文聚焦「现状 → 一期结构化」的落地。

## 现状（过渡态）
- **存储**：浏览器 `localStorage["pdfTranslateKnowledgeProfiles"]` = `{ [name]: 纯文本 }`。
- **提交**：翻译时把整段文本作为 `knowledge_base` 传给后端。
- **注入**：`default_translation_prompt` 把它整段拼进 `knowledge_block`，优先级高于文档上下文。
- **缺陷**：非结构化、无命中检索、不跨设备、无版本审校。

## 一期目标（结构化 + 服务端）
数据结构见 [公共设计/数据模型.md](../公共设计/数据模型.md) 的 `KnowledgeProfile`：`glossary[] / style_rules[] / do_not_translate[]`。

### 改动清单
1. **存储迁移**：`storage/knowledge/{name}.json`，新增 CRUD API（[HTTP-API.md](../接入规范/HTTP-API.md) 预留段）。
2. **命中注入**：翻译前用文档文本扫描 `glossary`，**只注入命中的术语**（降噪、提命中率）；`style_rules`/`do_not_translate` 全量注入。
3. **硬保护（可选）**：禁译词在送模型前做占位保护，回填原文，避免被改写。
4. **前端结构化编辑**：知识库视图从单个 textarea 升级为「术语表格 + 风格列表 + 禁译列表」。
5. **兼容导入**：旧 `{name, content}` 纯文本 → 整段进 `style_rules`，不丢数据。

### 质检联动
`stats` 增加 `glossary_hits`，质检报告「知识库」项显示「本次命中 N 个术语」。

## 与二期 MCP 的衔接
一期把知识**结构化**，二期才能把它**外置 + 检索**（见 [MCP接入规范.md](../接入规范/MCP接入规范.md)）。注入点不变（仍是 prompt 的 `knowledge_block`），只是来源从「本地结构化」变为「MCP 检索结果」，并保留本地回退。
